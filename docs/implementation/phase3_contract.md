# Phase 3 Assurance and Authoring Vertical Slice Contract

Status: implemented vertical slice and executable acceptance contract

Branch: `agent/phase3-assurance-authoring`

Accepted Phase 2 base: `6a14d52e4655dc8b7d5a42e43467f4c58faba510`

## 1. Outcome

Phase 3 is complete only when one central theoretical result can
move from an exact, fresh, human-confirmed G5 `ValidatedArgumentPackage`
through enhanced assurance, a thin Paper IR, a canonical-writer manuscript
unit, three isolated reviews, and an executable authoring-readiness closure.
The accepted vertical fixture may use any registered result archetype, but its
prose must still make the economically operative force, affected margin,
serious rival, separating evidence, and failure boundary reconstructible.

The first executable slice is deliberately one result block with its local
reader path, not a claim that the system can already compile a complete paper
or meet a journal's publication threshold.

```text
fresh G5 package
-> blind independent re-derivation
-> exact proof audit against the sealed re-derivation
-> symbolic/counterexample receipts with bounded evidentiary roles
-> Paper IR projection and minimal universal authoring profile
-> reader states, section contracts, and archetype ResultPacket
-> one canonical writer and one exact manuscript artifact
-> isolated formal-fidelity, economic-reader, and cold-reader reviews
-> canonical revision/integration when needed
-> zero critical fidelity defects plus passed retell/transfer/boundary probes
-> replayable authoring-readiness closure
```

Phase 3 is provider-neutral. Deterministic fixtures and a provider interface
exercise the writer boundary in CI. Provider selection, credentials, general
agent orchestration, and comparative model evaluation are not smuggled into
the canonical runtime.

### Implementation and acceptance record

The implementation extends the accepted Phase 2 gold ObjectStore chain rather
than replacing it with a mocked authoring-only fixture. The accepted central
claim has three exact proof obligations, each with its own blind
`ReDerivationRecord`; the resulting `AssuranceBundle` binds all three records,
their proof-audit reports, and an actual reproducible harness receipt. Typed
non-applicability can explain why a second harness family does not fit an exact
obligation, but it cannot make all headline assurance observationally empty.

The gold authoring suffix deliberately demonstrates repair rather than an
immediate pass:

```text
first manuscript unit
-> formal-fidelity pass
-> economic-reader failure + cold-reader near-transfer failure
-> blocked ReviewClosure + exact RevisionBrief
-> same canonical writer supersedes the unit
-> fresh formal, economic, and cold-reader runs
-> all thirteen closure checks pass
-> AUTHORING-READY-0.1
```

This chain uses the real immutable artifact store. Operational validation reads
the exact re-derivation transcript, proof-audit report, harness code/input/
output/receipt, manuscript, probe, sealed key, response, and revision-brief
bytes. It recomputes their hashes and structural bindings; a payload that merely
self-reports matching evidence is insufficient. Manuscript span offsets and
text hashes are checked against the stored UTF-8/LF manuscript, and the
canonical writer's role-packet and output hashes bind the actual compilation
boundary.

Cold-reader isolation is also operational: the writer context excludes probes
and keys; the respondent receives the frozen manuscript, declared reader
background, transfer objective, and visible probes but not the key; the
distinct adjudicator receives the exact sealed key and response. Separate
compartments and immutable context lineage make actor-label substitution
insufficient to bypass that separation. The chain then appends a real
`HumanEffortRecord` and verifies exact replay at the final head.

The comprehensive Phase 3 suffix currently incurs noticeable runtime cost
because repeated commits replay the growing chain and revalidate operational
lineage and artifact bytes. Performance optimization is therefore a recorded
follow-up; it may introduce safe caching or incremental validation only if the
same fail-closed acceptance result is preserved.

## 2. Compatibility before capability

- `registry.v1.json`, `registry.v2.json`, their instruction bundles, pinned
  hashes, and historical selectors remain frozen.
- Phase 3 introduces `registry.v3.json`. Existing scientific routes retain
  their prior route semantics; the three authoring routes receive executable
  contracts in the v3 catalog.
- Phase 1 canonical bytes remain byte-for-byte replayable.
- Phase 2 theory schemas, registered payload set, v2 policy bytes, and golden
  semantic closure remain frozen and receive a dedicated oracle.
- Phase 3 payloads use a separate authoring namespace and schema export. They
  do not acquire new meanings by being inserted into the Phase 2 theory
  registry.
- Historical v1 and v2 transactions resolve their exact policy by registry
  hash. Neither historical catalog is a live-write downgrade.
- Paper IR and manuscript projections are never scientific evidence for a
  `ClaimGraph`, `AssumptionMap`, proof, mechanism, or G1--G5 promotion.

## 3. Phase 2 verification versus Phase 3 assurance

Phase 2 establishes that exact obligations have admissible verification and
that a G5 argument package has a fresh, noncompensatory closure. Phase 3 adds
assurance about how that verification was obtained and how it may be rendered.

A blind `ReDerivationRecord` is created in a run whose context contains the
exact formal statement, model, assumptions, and obligations but excludes the
originating verification record, proof artifact, proposed intuition, and G5
package prose. A later audit run may see the sealed re-derivation together with
the original proof evidence. `AssuranceBundle` therefore records:

- the exact G5 package, claim graph, formal model, assumption map,
  verification bundle, and headline claim;
- at least one exact sealed `ReDerivationRecord` performed by an actor
  independent of the originating verifier of the headline obligation;
- proof audits bound to exact claim, obligation, assumption, model, proof
  artifact, and verification revisions;
- symbolic or counterexample harness receipts with tool/version/code/input/
  output/certificate-or-receipt hashes, exact domain, witness, outcome, and
  limits (plus a seed only for a future protocol that actually uses one);
- unresolved issues and whether any is critical.

The VAP is the assurance authority root, not merely another selected entity.
The selected `ClaimGraph`, `FormalModel`, `AssumptionMap`, and
`VerificationBundle` revisions must equal the four refs stored inside that
exact VAP. A current same-type substitute from another package fails even if
its local payload is otherwise valid. Likewise, an obligation or verification
record has assurance authority only when it is an exact member of that VAP's
`VerificationBundle`.

Finite examples, grids, simulations, and failure to find a counterexample can
corroborate or falsify. They cannot discharge a universal theorem. A symbolic
identity can discharge only the exact algebraic identity and domain covered by
its reproducible certificate; it does not silently prove equilibrium existence,
optimality, interpretation, or a stronger scope.

Raw tool transcripts and originating proof persuasion are assurance artifacts.
The canonical writer sees only the accepted statement, assumption roles,
economic argument, boundary, and reader-facing proof roadmap.

`ASSURANCE-PASS-0.1` is derived, never agent-authored. It requires a current
exact G5 closure; a sealed re-derivation whose actor, run, context, and parent
lineage are independent of the proof author and originating verifier; exact
proof-audit coverage; reproducible symbolic and counterexample harness
receipts (or a typed, reasoned non-applicability record for the exact check);
and no failed, falsified, inconclusive-critical, or unresolved-critical item.
Working/submission design and `AUTHORING-READY-0.1` both require this predicate.
A valid counterexample or critical gap records an exact blocker, prevents use
of the VAP handoff for authoring, and routes to theory repair. It cannot be
waived by favorable prose reviews.

The blind record binds its exact `RouteRun`, `ContextManifest`, compiled-context
hash, derivation artifact, actor, model/tool lineage, and declared ancestor
runs. Different actor IDs with a shared prohibited context or proof-derived
ancestor do not count as independent.

The Phase 3 suite exercises both the exact symbolic-identity harness and the
finite counterexample harness. For one particular claim, either family may be
`not_applicable` only with a typed reason tied to its obligation. Every receipt
must be reproducible from recorded tool/code/input/domain/seed versions;
tampering with any of them or with the certificate/output must fail replay or
assurance validation.

This establishes byte, execution, and declared-reference integrity for the
harness predicate. Phase 3 does not mechanically establish that an executable
predicate chosen by an author or agent is semantically equivalent to the full
natural-language proof obligation. A typed, reviewable obligation-to-predicate
mapping and adversarial checks against vacuous or weakened encodings are an
explicit Phase 4 hardening boundary. Until then, a receipt cannot carry a
broader evidentiary role than the exact predicate it executes.

## 4. Canonical authoring objects

Phase 3 adds these typed projection and review entities:

| Entity | Function |
|---|---|
| `ReDerivationRecord` | blind, sealed independent derivation bound to exact formal inputs |
| `AssuranceBundle` | exact independent re-derivation, tool-check, and proof-audit closure |
| `ResolvedProfileManifest` | minimal universal-floor projection and exact profile artifact ref |
| `PaperIR` | thin claim/ontology/narrative/profile projection of one exact package |
| `ReaderPath` | sparse reader-belief states and ordered section contracts |
| `ResultContractSet` | archetype-sensitive ResultPackets, assumption and proof-roadmap contracts |
| `CriticAssignment` | one sealed role/context contract for one independent critic |
| `ManuscriptUnit` | exact artifact, canonical-writer ownership, spans, and prose realization map |
| `ReaderProbeSet` | post-freeze retell, scope, boundary, and transfer probes plus a sealed key |
| `ReaderResponse` | a cold reader's answers produced without access to the sealed key |
| `ReviewFinding` | one immutable consequential finding with exact target, evidence, severity, and disposition |
| `ReviewRecord` | one critic's immutable findings and role-specific tests |
| `ReviewClosure` | deterministic `AUTHORING-READY-0.1` evaluation over exact reviews |
| `RevisionBrief` | typed unresolved findings returned to the canonical writer |
| `HumanEffortRecord` | append-only active-time and semantic edit events |

Paper IR caches source fields only with their exact entity version and JSON
Pointer. It cannot be independently edited as a source. Every projection is
rebuildable from current scientific state plus effective human Decisions.
The rich IR, reader plan, and contract document are immutable artifacts. Their
canonical entities remain thin identities, hashes, exact source mappings, and
the minimum structural projections needed for validation; they never copy the
full manuscript, proof, tool transcript, or research archive into the snapshot.
`ReviewRecord` is a manifest over exact granular `ReviewFinding` entities, not
a second nested finding store.

## 5. Minimal universal authoring profile

Phase 3 closes the dependency on the deferred profile system with one minimal,
versioned `ResolvedProfileManifest` artifact/ref consumed by Paper IR:

```text
universal-floor version
+ one human-selected theory mode
+ one ambition
+ one primary result archetype
+ one primary audience
```

The relevant exact Decision refs are recorded and must remain effective for a
working or submission compilation. Theory mode, ambition, and audience use
their existing L2 Decision kinds. The primary archetype is projected from the
exact G4/G5 result hierarchy rather than decided a second time. Field profiles,
venue overlays, submission constraints, craft corpora, retrieval, and anchor
selection remain Phase 4.

The minimum floor requires:

- benchmark before notation or result;
- stable economic identity before first inferential use of a symbol;
- exact theorem plus an entailed economic translation;
- archetype-specific economic or conceptual logic, not a vague restatement;
- a diagnostic example, ablation, separator, or witness;
- assumptions explained where economically operative;
- a boundary and a proof roadmap;
- explicit result hierarchy and appendix discipline;
- no workflow, schema, route, gate, entity-ID, or prompt vocabulary in prose.

## 6. Paper IR and reader contracts

`PaperIR` binds:

- compiler mode: `preview`, `working`, or `submission`;
- exact package and G5 Decision when applicable;
- source state revision and upstream projection hash;
- minimal profile and exact Decision refs;
- claim projections with allowed strength, scope, assumptions, locations, and
  prohibited extensions;
- economic ontology with stable names, symbols, roles, aliases, and first-use
  requirements;
- narrative spine;
- one canonical writer actor.

Paper IR may project the G5-approved contribution spine and make reversible
local ordering choices at L1. Changing the central reader update, main-result
hierarchy, or material argument order requires an exact effective
`argument_spine` or `narrative_material_order` human Decision before working
compilation.

`preview` may use a bounded provisional package, may omit assurance, and is
visibly labeled. The exact `PaperIR.mode` supplies the begin-time mode contract
for composition; preview can never produce
`ReviewClosure.status=authoring_ready`. `working` and `submission` consume the
exact fresh package approved by the effective G5 Decision. Submission is a
second local compilation after a working unit has a fresh authoring-ready
`ReviewClosure` and a human `manuscript_version_promotion` Decision whose
subject is that exact closure. The closure binds the exact manuscript unit and
artifact hash, preventing reuse of an old promotion after a revision. This is
not external release or submission execution. The local submission compilation
may change only formatting and the resulting span offsets. It must preserve the
exact ordered assertions, span text hashes, typed roles, source bindings,
claim/scope/assumption semantics, and governing working-unit lineage; any
wording or scientific-content change must return through working composition
and fresh review.

`ReaderPath` is an ordered DAG. Each required knowledge item must be delivered
by an earlier state or declared as target-audience background. Circular
motivation, unexplained prerequisites, and definitions after inferential use
are rejected.

Every central result has one `ResultPacket` with the shared layers:

```text
question
pre-result expectation or reasoned not-applicable
formal statement and scope
economic translation
archetype-specific explanation
boundary
proof roadmap
consequence
```

For `mechanism_explanation`, the packet additionally binds the initiating
force, affected margin, rival, separating example, ablation, and failure case.
Other archetypes retain typed schema contracts but are not claimed as Phase 3
runtime acceptance cases.

Coverage is per packet, never inferred from a manuscript-level pass. Each
`ResultPacket` must have its own economic reconstruction bound to the exact
claim projection and packet, and its own cold-reader coverage for retell,
scope, assumption role, boundary, and near transfer. Strong coverage of one
headline packet cannot compensate for a missing or weak second packet.

An example passes only when it performs its registered functional role--for
example benchmark, mechanism-on, rival separator, ablation, boundary,
necessity, tightness, or representation witness--and its exact evidence supports
that role. The validator does not require every legitimate example to change a
primitive or separate a rival.

## 7. Prose realization and entailment

Every consequential manuscript span has a stable assertion ID and exactly one
role:

- `formal_statement`;
- `economic_translation`;
- `mechanism_or_conceptual_explanation`;
- `example_or_witness`;
- `assumption_interpretation`;
- `boundary`;
- `proof_roadmap`;
- `literature_comparison`;
- `consequence`;
- `conjecture`.

The span records its exact claim/version, source field paths, scope,
assumptions, support refs, manuscript location, and one proposed value from the
closed wording lattice `exact`, `entailed_equivalent`, `entailed_weaker`, or
`explicit_conjecture`. This label is not self-certifying. The isolated
formal-fidelity critic produces an exact `EntailmentCheck` for every
consequential assertion, recording the scope relation and conclusion relation.
Only `equal/equivalent` or a verified subset/weaker relation is admissible;
stronger, incomparable, unsupported, or inconclusive language blocks closure.
Displayed formal statements must preserve a normalized authoritative content
hash covering conclusion, scope, quantifiers, and assumptions; harmless
typesetting or naming changes need not be byte-identical. Mechanism
explanations require argument, example, ablation, or witness support.
Consequences and conjectures cannot be rendered as theorem implications.

Typed coverage is exhaustive over the manuscript body: every non-whitespace
character must fall inside exactly one registered prose span. Headings, connective
sentences, qualifications, and conclusions cannot form an untyped channel for
unsupported claims. Whitespace separators may remain outside spans; overlapping
or uncovered substantive text fails validation.

Every source field receives a field-level invalidating relation to the exact
manuscript unit. A theorem-scope change therefore stales every dependent or
stronger span and its reviews while leaving units bound only to unchanged
claim fields fresh.

System-language scanning uses the versioned narrow rule
`AUTHORING-LEAKAGE-LINT-0.1`: exact route IDs, G1--G5 internal labels,
entity/relation ID patterns, schema identifiers, context-manifest keys, prompt
directives, and named hidden-evaluation artifacts. Ordinary economic uses of
words such as claim, proof, model, scope, or mechanism are never banned.

## 8. Route catalog v3

All ten Phase 3-native routes enforce exact selected-input closure in the
immutable run focus. Registry type and cardinality rules are necessary but not
sufficient: the focused entity IDs and revisions must be exactly those bound by
the transaction's output payload, artifact lineage, and governing upstream
objects. An otherwise valid same-type object from another authoring chain--
including a foreign `PaperIR`, `ResultContractSet`, probe, response, review,
closure, or `AssuranceBundle`--is rejected rather than silently substituted.

The immutable run focus and the provider-visible context have different jobs.
The focus's set of current `EntityVersion` refs must equal the transaction's
complete set of `EntityVersion` evidence--no omitted authority input and no
extraneous focused entity. The provider-visible selected refs and role packet
are a privacy projection of that focus. A blind re-derivation may therefore
exclude the VAP from provider-visible context while the immutable focus still
retains the VAP as an entry and authority binding.

### `verify.independent_rederivation`

Consumes the exact package only as an entry/authority binding and compiles a
blind packet from the current claim, model, assumptions, and one selected proof
obligation. The unique selected obligation and its unique originating
`VerificationRecord` must be an exact pair in the package's
`VerificationBundle`. The writer's explanation, originating verification
records, proof artifacts, and VAP prose are excluded. It produces exactly one
sealed `ReDerivationRecord`; a failure or counterexample is preserved rather than
massaged into a successful derivation. Each run must select exactly one
`ProofObligation`, and the record must bind that same exact obligation revision;
batching several obligations or substituting a same-type foreign obligation is
invalid.

### `audit.argument_assurance`

Consumes the sealed re-derivation plus the exact package, verification bundle,
verification records, proof artifacts, model, assumptions, and obligations. A
different independent actor performs the proof-revision audit and reproducible
symbolic/counterexample checks, then produces exactly one `AssuranceBundle`.
Audit coverage is a same-package bijection: every exact `VerificationRecord`
in the package bundle has exactly one re-derivation bound to that record and
its obligation, and every included re-derivation maps back to exactly one such
record. Missing, duplicate, orphaned, or cross-package mappings fail closed.
A critical gap records a blocker and prevents authoring; it does not silently
rewrite or retain the affected scientific claim. Repair returns to the theory
routes.

### `design.reader_path`

Consumes one exact package, its current authoring inputs, and--for working or
submission mode--one passing `AssuranceBundle`. Preview may omit assurance but
can never close as authoring-ready. The route produces exactly one `PaperIR`,
`ResolvedProfileManifest`, `ReaderPath`, and `ResultContractSet`, plus exactly
three `CriticAssignment` entities for:

1. `formal_fidelity`;
2. `economic_reader`;
3. `cold_reader`.

The assignments name distinct actors, allowed information, forbidden context,
and the transfer objective. They contain no concrete cold-reader probe,
answer-key criterion, expected answer, gold rewrite, or another critic's
finding.

### `compose.manuscript_unit`

Consumes the exact design outputs and package. Produces one manuscript artifact
and one `ManuscriptUnit`. The transaction actor must equal the Paper IR's
canonical writer. An initial composition has no prior unit. A revision must
consume one exact current unit and one fresh `RevisionBrief`, supersede the
unit, preserve accepted spans where possible, and resolve or explicitly carry
every finding. The manuscript unit must realize all applicable central layers
and bind each consequential span to exact sources. It may not change scientific
scope or decisions.

If the registered manuscript artifact has been edited outside the compiler,
hash mismatch blocks composition. The canonical writer may register the human
version or propose a three-way reconciliation, but it cannot overwrite those
bytes.

### `review.manuscript_unit`

One assigned `formal_fidelity` or `economic_reader` critic produces one
`ReviewRecord` plus zero or more immutable `ReviewFinding` entities and cannot
write prose. Actor, assignment, and context must agree. Neither critic may be
the canonical writer, and the two actors are distinct.

### `prepare.reader_probe`

After a manuscript unit is frozen, the cold-reader assignment's probe designer
creates one `ReaderProbeSet`. Its concrete probes and exact answer key are
registered as separate artifacts. Neither artifact may have entered any writer
context, and the answer-key bytes are sealed from the respondent.

### `answer.reader_probe`

The assigned cold respondent receives the manuscript unit, declared
target-reader background, transfer objective, and probe artifact--but not the
answer key, authoring rationale, gold prose, or other critics' findings--and
produces one `ReaderResponse` artifact/entity.

### `adjudicate.reader_probe`

The assignment's third, distinct adjudicator receives the sealed key and exact
response and produces the cold-reader `ReviewRecord`. The record therefore
cannot self-report a pass without an inspectable probe/key/response lineage.

### `close.manuscript_review`

A deterministic closure actor explicitly consumes the exact current
`AssuranceBundle` governing the manuscript chain plus exactly one fresh
formal-fidelity, economic-reader, and cold-reader review. Omitting assurance or
selecting a passing bundle from another package/unit chain fails closed. The
route produces one `ReviewClosure` and, when any repairable finding blocks
readiness, one exact `RevisionBrief`.

### `record.human_effort`

A human actor appends one `HumanEffortRecord` version. Combining these distinct
run shapes into optional outputs of `review.manuscript_unit` is forbidden,
because begin-time actor and context isolation would become ambiguous.

## 9. `AUTHORING-READY-0.1`

A working manuscript unit is authoring-ready only when all of these hold:

`Current and fresh` is a transitive whole-chain condition. It covers the G5
Decision and package; assurance bundle, re-derivations, audits, harness records,
and exact scientific inputs; profile, Paper IR, reader path, result contracts,
and critic assignments; manuscript artifact and field-level dependencies; all
review findings and records; and the probe, sealed key, response, adjudication,
and closure. Superseding or staling any member invalidates readiness even when
the terminal closure bytes are unchanged.

1. its exact G5 package and minimum profile Decisions are current and fresh;
2. its exact current AssuranceBundle satisfies `ASSURANCE-PASS-0.1`;
3. every non-whitespace manuscript-body character belongs to exactly one typed
   span, and every span traces to exact current scientific sources;
4. all applicable layers of every ResultPacket appear in actual prose spans;
5. no span exceeds the authoritative claim scope or drops a required
   assumption;
6. no numerical or finite check is described as a universal proof;
7. the formal-fidelity review has zero unresolved critical findings;
8. for every ResultPacket, the economic-reader review confirms a bound,
   reconstructible, non-restatement explanation and diagnostic example/witness;
9. for every ResultPacket, the cold reader passes question/benchmark retell,
   exact-scope recovery, assumption-role recovery, boundary discrimination,
   and near transfer;
10. the reader-state DAG is closed and symbols have stable first-use economic
   identities;
11. no system-governance vocabulary or hidden probe material appears;
12. canonical integration has one coherent terminology and result hierarchy;
13. every blocking finding is resolved; if resolution requires an L2 choice,
    the exact effective human Decision must exist and govern the current unit.
    Merely labeling a finding "needs human decision" remains blocking.

`ReviewClosure` is a derived validation record, not a human promotion and not
evidence of publication quality.

The bounded repair loop is executable and provenance-preserving:

```text
compose
-> isolated reviews and reader adjudication
-> closure plus RevisionBrief when blocked
-> canonical-writer supersession
-> fresh independent reviews
-> closure
```

## 10. Human effort

`HumanEffortRecord` is append-only and records active minutes, affected unit,
edit disposition, severity, and one of:

- formal correction;
- mechanism/intuition repair;
- assumption interpretation;
- reader path/result hierarchy;
- literature positioning;
- voice/language;
- formatting only;
- deletion or scope reduction.

Unattended compute time is excluded. A superseding version must preserve every
prior event byte-for-byte and append only later events.

## 11. Required adversarial tests

- stale or unconfirmed G5 cannot produce working/submission output;
- every Phase 3-native route rejects same-type foreign selected inputs even
  when registry type and count constraints still pass;
- immutable run focus rejects omitted or extraneous current `EntityVersion`
  evidence, while the blind provider-visible packet may exclude the VAP;
- assurance rejects a selected `ClaimGraph`, `FormalModel`, `AssumptionMap`, or
  `VerificationBundle` that differs from the exact refs inside its VAP;
- independent re-derivation rejects zero, multiple, or foreign
  `ProofObligation` inputs, or an obligation/verification pair absent from the
  package's exact `VerificationBundle`;
- assurance audit rejects missing, duplicate, orphaned, or cross-package
  re-derivations instead of a one-to-one mapping over `VerificationRecord`s;
- review closure rejects omitted or foreign `AssuranceBundle` input;
- preview cannot receive an authoring-ready closure;
- same-actor re-derivation and proof audit are rejected;
- different actor IDs with shared prohibited proof/context lineage are rejected;
- finite search cannot prove a universal claim;
- symbolic and finite harness receipts reproduce exactly, and tampering fails;
- proof audits with mismatched claim/model/assumption/proof revisions fail;
- Paper IR projections cannot be cited as scientific support;
- introductions or conclusions stronger than the body claim fail;
- missing assumptions and numerical-as-proof language fail;
- reader prerequisites and cycles fail;
- theorem paraphrase without explanatory delta fails the economic-reader gate;
- examples that cannot perform their claimed registered functional role fail;
- missing boundary, late symbol identity, or terminology drift fail;
- untyped non-whitespace manuscript text and overlapping typed spans fail;
- manuscript-level economic/cold passes cannot hide an uncovered ResultPacket;
- workflow/schema/gate vocabulary in prose fails;
- pure-theory non-mechanism schema examples are not forced to invent welfare,
  policy, direct effects, or equilibrium feedback;
- writers receive no hidden probes, case keys, gate logs, or raw tool
  transcripts;
- external human manuscript edits enter a hash-mismatch/reconciliation path and
  are never overwritten;
- theorem-scope mutation stales dependent prose/reviews/closure but not an
  unrelated manuscript unit;
- submission compilation rejects wording, assertion, source-binding, or
  scientific-content changes and permits formatting/offset changes only;
- staling any readiness dependency invalidates an otherwise unchanged closure;
- human effort versions are append-only;
- exact replay, Phase 1 frozen bytes, and Phase 2 frozen semantics pass.

## 12. Explicit deferrals

Phase 3 does not implement:

- full field/audience/venue profiles, craft retrieval, or paper anchors;
- a complete-paper compiler or all six archetype runtime cases;
- general multi-agent panels, voting, judges, or broad orchestration;
- Lean or another formal-proof adapter;
- provider credentials, provider ranking, or held-out model comparison;
- semantic certification that a harness predicate fully represents its
  natural-language proof obligation; this is a Phase 4 hardening boundary;
- release bundles, external communication, external submission execution, or
  L3 handoff;
- any empirical, econometric, identification, estimation, or data workflow.

## 13. Exit criteria

The implemented Phase 3 vertical slice is accepted against these criteria:

- registry v3 enables the two assurance routes, three core authoring routes,
  and five narrowly scoped reader/closure/effort child routes without changing
  v1/v2 bytes;
- all Phase 3 payload schemas are exported and strict;
- one real ObjectStore chain continues the accepted Phase 2 gold case through
  blind re-derivation, proof assurance, design, compose, formal/economic
  reviews, sealed probe/response/adjudication, closure, exact replay, and one
  human effort append;
- a central result is traceable from prose to claim, assumptions, mechanism,
  diagnostic example, proof status, and exact evidence;
- the cold-reader retell, boundary, and near-transfer tests actually pass;
- a scope mutation demonstrates selective prose invalidation;
- deterministic good and adversarial manuscript fixtures test function rather
  than preferred wording;
- one blocking fixture produces a RevisionBrief, is superseded by the same
  canonical writer, and passes only after fresh independent re-review;
- the full Phase 1--3 regression suite, schema checks, doctor, and frozen
  oracles pass.

Passing this contract establishes a trustworthy authoring core. It does not
establish Econometrica/Top-5 quality, comparative superiority to v1, or reduced
human effort on held-out papers. Phase 4's full profile and craft system remains
unimplemented, and comparative quality and efficiency claims require the
held-out work in Phase 6.
